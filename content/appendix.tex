\section{Computational Efficiency}
\label{subsec:throughput}

To evaluate the computational efficiency of our data generation pipeline, we measured the throughput using a model system consisting of an uncapped tetra-alanine peptide (AAAA). 
The fully solvated system comprised 1,682 atoms in total (of which 43 atoms were from the solute) surrounded by explicit water molecules with a 1.2~nm padding. 
All benchmarking runs were executed on a single NVIDIA A100 GPU.

We benchmarked two parallelization strategies and compared their throughput as a function of the number of concurrently running systems (Figure~\ref{fig:throughput_combined}).

The first strategy consists of conventional independent simulations at a constant temperature of 300~K. To maximize GPU utilization, we simulated multiple independent copies of the system concurrently, dispatching each run in isolation via its own dedicated system process.

The second strategy employs \gls{remd} \cite{qi2018replica}, which runs simulations at multiple temperatures and periodically exchanges configurations between replicas to improve conformational sampling. This approach helps avoid biased datasets that overrepresent only a limited set of conformations. For the \gls{remd} protocol, we leveraged the \texttt{reform} package \cite{chen_noe_reform} to orchestrate replica swaps. By default, the multi-threaded interface of \texttt{reform} does not support periodic boundary conditions. However, modeling the explicit solvent requires periodic boundary conditions to avoid surface artifacts. To overcome this limitation, we constructed a custom wrapper around \texttt{reform}'s single-threaded API, managing a dedicated thread per replica to parallelize execution while retaining support for periodic boundary conditions.

\subsection{Solvent Scaling with Sequence Length}
\label{subsec:sequence_length}

Because a fixed margin of water must surround the peptide in all three dimensions to avoid boundary artifacts, the total volume and therefore the number of solvent atoms scales cubically ($\mathcal{O}(L^3)$) with the sequence length $L$. This rapid addition of solvent atoms makes large-scale simulations of long peptides slow (Figure \ref{fig:system_scaling}).

\end{figure}

\section{Evaluation of Alternative Dataset Architectures}
\label{sec:appendix_dataset_architectures}

Prior to adopting the sharded \texttt{.npy} architecture detailed in the main text, we evaluated several standard data-loading paradigms. These approaches ultimately proved inadequate for our specific constraints---namely, out-of-core dataset sizes and highly concurrent distributed training. We detail these iterative attempts and their failure modes below.

\paragraph{1. PyTorch Native Workers with Global Shuffle}
Our initial baseline utilized PyTorch's native \texttt{DataLoader} workers coupled with a standard global shuffle. This approach resulted in severe disk I/O bottlenecks. In this context, the primary performance barrier is the disk-to-RAM transfer bandwidth, rather than L1/L2/L3 CPU cache hierarchies. Global shuffling induces scattered, randomized disk reads, completely negating spatial locality. Ideally, once a data page is loaded into system memory (RAM), subsequent samples from that page result in fast memory hits. However, because our dataset significantly exceeds available RAM capacity, the global randomized access pattern led to severe page thrashing. ``Hot'' pages were continuously evicted from memory before their contents could be sufficiently utilized, resulting in a continuous loop of expensive disk fetches.

\paragraph{2. SQLite-Backed Dataset}
To manage the out-of-core data, we subsequently implemented a SQLite-backed relational database. While this approach conveniently abstracted away complexities related to tensor alignment and padding, it yielded poor overall training performance. The SQLite implementation suffered from the exact same page thrashing and poor cache locality as the naive PyTorch approach, further compounded by a significantly higher baseline memory overhead.

\paragraph{3. LMDB with Custom Windowed Sampling}
To explicitly leverage OS-level page caching, we transitioned to a Lightning Memory-Mapped Database (LMDB). To mitigate the severe thrashing observed in earlier attempts, we engineered a custom windowed sampler and memory manager. This system restricted dataset sampling to a localized ``window'' guaranteed to fit within the system's available RAM, ensuring that active pages remained hot.

While this LMDB approach achieved near-optimal throughput for isolated, single-GPU training runs, it introduced a critical bottleneck during distributed hyperparameter sweeps. When multiple independent GPU jobs concurrently accessed the shared LMDB file, the resulting localized access patterns from different jobs severely fragmented disk I/O. This concurrent access crippled the underlying disk throughput, drastically slowing down training across the cluster.

\paragraph{Conclusion}
The sequential failure of these architectures to handle multi-GPU concurrency motivated the design of our final sharded \texttt{.npy} pipeline. By decoupling static metadata, parallelizing reads across randomized shards, and utilizing asynchronous prefetching, the final approach naturally mitigates concurrent I/O contention and scales seamlessly to clusters of 20 or more GPUs.
